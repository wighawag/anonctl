# The `/etc/anonctl/<account>.json` marker: schema, precedence, and trust framing

anonctl exposes a versioned JSON marker at `/etc/anonctl/<account>.json` (`internal/marker`) that sibling tools (anon-pi, netcage) read to detect "this account is already kernel-anonymized" and skip re-forcing a proxy (the Tor-over-Tor / double-anonymization guard, stories 28/29). We decided its shape and rules deliberately because it is a CROSS-TOOL WIRE CONTRACT (hard to reverse once other tools parse it) and several of its choices are surprising without this context.

The record carries EXACTLY: `schemaVersion` (int, starts at 1, additive-only evolution, a newer version is REFUSED by `Parse` rather than mis-read), `account`, `uid`, `endpointClass` (`tor-shared` / `socks-peruser`, the ONE piece of endpoint detail a consumer needs, story 20), `createdAt` (RFC3339 UTC), and `anonctlVersion`. It DELIBERATELY EXCLUDES the endpoint URL and any credentials: the file is world-readable under `/etc` (dir 0755, file 0644, chosen so any UID sibling tool can read the dependency-free signal), so it must never hold a secret. This is the same credential-free-at-rest hygiene as the endpoint model (ADR 0001); a consumer needs only "forced + which share-class", which `endpointClass` gives.

Detection PRECEDENCE, from authoritative to hint: (1) the marker FILE is the authoritative, dependency-free signal, a consumer reads `/etc/anonctl/<account>.json` with NO anonctl binary needed; (2) `anonctl status --json` is a CONVENIENCE READER of that same file (`AccountStatus.Forced` / `.Marker`), not a second source of truth; (3) the `anon` / `anon-<name>` name prefix is a HINT only, never authoritative (a same-named account with no marker is not forced).

The marker is written STRICTLY AFTER `verify` passes (`Store.WriteVerified` refuses on a non-passing verify with `ErrVerifyNotPassed`) and removed on `rm`/teardown, because it is a COORDINATION CLAIM, not a live security proof. Writing it for an unverified account would be a false claim. A marker-write failure does NOT flip a passing verify to a failure (the account IS proven forced); a consumer needing certainty runs `anonctl verify` or its own leak check rather than trusting the claim.

The `/etc` write is a SHARED/GLOBAL location, so the base directory sits behind a configurable lever (`Store.BaseDir`, default `/etc/anonctl`): production uses the real path; the unit tests point it at a scratch temp dir and assert the real `/etc/anonctl` is left untouched (the shared-write isolation discipline), so the whole (de)serialization + schema-version + write-after-verify path is proven with no root and no real `/etc` write.

Considered and rejected: (a) embedding the endpoint URL/creds so a consumer could re-dial it (rejected: world-readable secret, and a consumer only needs the share-class, not the endpoint); (b) treating a newer `schemaVersion` as best-effort-parseable (rejected: acting on partially-understood fields of a future marker is worse than a loud refusal); (c) making the name prefix authoritative (rejected: it is trivially spoofable and says nothing about whether forcing is actually applied and verified).
