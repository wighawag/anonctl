# Reboot persistence: templated per-account shim unit, an anonctl-owned early loader, a standing default-deny, and the boot invariant

Status: accepted (amended: the boot invariant now holds by a standing default-deny + anonctl's own loader, not an nftables.service drop-in; and `rm` teardown now STOPS the shim unit BEFORE it userdels the shim account)

## Context

An anon account's forcing must SURVIVE a reboot and re-apply FAIL-CLOSED, with no window where the anon UID has un-anonymized egress at boot (the BOOT INVARIANT: "at no point during boot does the anon UID have direct egress"). anonctl applies everything itself as root (the ufw stance) and does NOT own the endpoint's lifecycle. This task (`persistence-and-boot-invariant`) chose the persistence mechanism and the unit shape; several decisions are load-bearing and would surprise a future reader, so they are recorded here.

**CORRECTION (from the e2e finding, BUG 1).** The original decision below claimed the boot invariant "holds by construction because nftables.service loads early." That was FALSE on a default Debian host: anonctl persisted its ruleset via a drop-in on the host's `nftables.service`, but Debian ships that service DISABLED and anonctl never enabled it, so after a real reboot the forcing rules were ABSENT and the anon UID egressed with the host's real public IP in the clear (`{"IsTor":false,"IP":"51.7.210.66"}`). Chasing the invariant by "loading the allow-through-shim rules early enough" was the WRONG invariant: it meant "un-forced" could mean "free" during any window the rules were not (yet) loaded. The mechanism was inverted and the loader replaced; see the amended decisions.

## Decisions

- **The per-account shim is a systemd `@`-template unit (`anonctl-shim@<account>.service`), NOT one multiplexer process for all accounts.** The per-account process boundary IS the security boundary: each account's shim runs under its OWN dedicated shim UID (closure b: only the shim UID may reach the endpoint), so a single multiplexer would collapse that boundary. systemd's `@`-template gives every account its own supervised instance from ONE unit file; `add` runs `enable --now`, `rm` runs `disable --now`, `update` runs `restart` (to pick up a rewritten endpoint). One template file, N independent instances, N distinct shim UIDs.

- **The template unit is account-AGNOSTIC; per-account parameters come from a per-instance `EnvironmentFile` (`/etc/anonctl/shim/<account>.env`).** `%i` is the account (the instance), and the shim UID, loopback ports, endpoint address, and derived isolation username are read from the env file, so ONE template serves every account. The unit starts as root only long enough to `setpriv --reuid ${ANONCTL_SHIM_UID}` down to the account's dedicated shim UID (exactly as the validated recipe runs the shim); the shim itself never runs as root. `User=` was rejected because it cannot read a per-instance env var, and baking the UID into the unit would break the single-template property.

- **Unit ordering is `After=network.target` only; the shim deliberately does NOT `Wants=`/`After=` the endpoint's own service.** anonctl does not own the endpoint lifecycle (it does not manage `tor.service`), and the nft rules FAIL CLOSED (drop) if the endpoint is not yet up, so there is no leak window to order against: the worst case is dropped-until-shim-and-endpoint-are-up, never leaking. `add` prints a reminder to enable the endpoint (e.g. `tor.service`) at boot, since that is the operator's job.

- **The boot invariant is inverted: the anon UID's RESTING STATE is a standing per-UID DEFAULT-DENY, and forcing only ever OPENS the shim path.** Each account has a tiny, standalone `inet` table (`anonctl_baseline_<account>`, `internal/nftables.GenerateBaseline`), SEPARATE from the per-account forcing table, whose sole job is: for the anon UID, RETURN loopback destinations and DROP everything else (all real, non-loopback egress, v4 AND v6). It is applied at `add`-time BEFORE the forcing rules and the shim, and persisted as its own always-loaded `<account>.baseline.nft`. The precise invariant it guarantees: **an anon UID with no anonctl forcing loaded is DROPPED, not free; forcing only ever OPENS the shim path.** If the forcing rules fail to load, the shim is down, or the endpoint is down, the account is STILL dropped, by construction. This is strictly safer than "load the allow-rules early enough."

- **Why this layers WITHOUT dropping forced traffic (the load-bearing nftables semantics).** A `drop` verdict is TERMINAL across all base chains at a hook; an `accept` is NOT (a later chain can still drop). So the baseline cannot simply drop the anon UID (that would kill forced traffic too). Instead, forcing's nat/output chain runs at `priority dstnat` (-100), BEFORE every filter chain, and REDIRECTS the anon UID's egress to a LOOPBACK shim port (rewriting the destination). The baseline (a filter/output chain, policy ACCEPT so it never touches another UID) RETURNs loopback (handing forced/shim traffic on to the forcing table's own closures) and DROPs everything non-loopback. Forcing present => the anon UID's traffic is redirected to loopback, the baseline returns it, the forcing table governs it (shim path works). Forcing absent => no redirect, the real egress stays non-loopback, the baseline DROPS it.

- **The nftables ruleset is loaded at boot by anonctl's OWN early loader unit (`anonctl-nftables.service`), NOT a drop-in on the host's `nftables.service`.** The unit `nft -f`s anonctl's per-account rule files (both the baseline and the forcing tables) from `/etc/anonctl/nftables/*.nft`; it is `WantedBy=sysinit.target`, `DefaultDependencies=no`, `Before=network-pre.target`, so it loads BEFORE the network is up and does NOT depend on the host's `nftables.service` (which Debian ships DISABLED and a later `systemctl disable nftables` could silently re-disable). anonctl OWNS this unit, so `add` enabling it mutates no host-owned service; `rm` on the LAST account disables it. The loader iterates the glob, so an empty/absent rules dir is a clean no-op and boot never fails when no account is forced. This REPLACES the earlier drop-in, whose reliance on a host-owned, silently re-disableable service caused the reboot leak. `add` still applies the LIVE rules itself via `nft`, so the unit only needs to fire at the NEXT boot (it is enabled without `--now`).

- **The boot invariant is proven by a reboot-EQUIVALENT early-boot simulation, not a real reboot.** The integration tests load the persisted rules the loader would `nft -f` at boot (the baseline AND the forcing table) with the shim NOT running, reproducing the boot window, then assert AS the anon UID that a direct outbound connection is DROPPED. They also prove BOTH directions of the inversion: forcing-ABSENT (only the baseline loaded, or the forcing table flushed) => DROPPED (the new guarantee, and the exact post-reboot state the finding observed leaking), and forcing-PRESENT => the redirect-into-shim path is intact (the baseline does not drop the redirected loopback traffic). They isolate to throwaway account/tables + a planted sentinel and assert the host's rules are untouched. The DEFINITIVE proof remains a real-host reboot: a human re-run of the e2e validation's step 5, PLUS a "flush the forcing table, confirm still dropped" check, is the final gate.

## Teardown ordering (amended from the e2e finding, BUG 1)

**CORRECTION (from `work/notes/findings/e2e-binary-revalidation.md`, PART E / BUG 1).** `add` and `rm` mutate two things per account: the login + dedicated shim UNIX ACCOUNTS (`internal/provision`, via `useradd`/`userdel`) and the FORCING artifacts (the nft tables, the persisted files, and the per-account shim UNIT `anonctl-shim@<account>.service`, via `internal/forcing` + `internal/systemd`). These are removed by two different calls, so their ORDER is load-bearing on teardown and is fixed here.

- **`rm` disables --now the shim unit BEFORE it userdels the shim account.** The dedicated shim service account (`<account>-shim`) is the UID the `anonctl-shim@<account>.service` unit runs as. `userdel` REFUSES ("user is currently used by process", exit 8) while any process runs as that UID. The original `runRm` called `provision.Rm` (userdel) BEFORE `forcing.Remove` (which `systemctl disable --now`s the shim), so on the LAST account a single `anonctl rm --purge-account` aborted at the userdel with the shim still running, and the entire last-account cleanup (nft tables, marker, `/etc/anonctl`, the shared template + loader units) never ran, leaving exactly the residue this ADR's cleanup was meant to prevent. The correct, now-enforced order is: **disable --now the shim unit -> remove the nft tables + config + marker -> userdel the login + shim accounts -> (last account) remove the shared units + empty dirs.** `forcing.Remove` already disabled the shim first internally; the fix was to sequence `forcing.Remove` BEFORE `provision.Rm` in `runRm` (they are the two seams, so the order lives at the caller). The ordering is asserted at the `runRm` seam by a unit test (the disable-shim call is recorded before the shim userdel) and proven no-residue by an `integration`-tagged teardown against a throwaway account whose shim UID is genuinely held by a running unit, with the host's other nft rules asserted untouched.

- **Teardown is proceed-and-report, not abort-on-first-error.** `runRm` runs every teardown step (remove forcing, userdel, remove marker) even if an earlier one failed, surfacing the first error with a non-zero exit. A half-torn-down account that silently stopped on the first error is worse than a fully-attempted, reported partial: the correct ordering makes the common path complete cleanly, and proceed-and-report makes an unexpected mid-teardown failure leave the least residue it can while still failing loudly.

## Consequences

Persistence introduced a per-account at-rest config (`internal/accountconfig`, `/etc/anonctl/accounts/<account>.json`, root-only 0600) as the source of truth the boot re-apply and `update` read; it is deliberately SEPARATE from the world-readable, credential-free marker (it may carry the endpoint address the marker must never hold). The `update`/`reconfigure` verb re-applies the nft rules (an atomic table replace: the default-DROP is never absent) BEFORE restarting the shim, so a reconfigure has no un-anonymized window. `add` gained an `--endpoint` flag (default: the local Tor SocksPort); `update`/`reconfigure` require it.
