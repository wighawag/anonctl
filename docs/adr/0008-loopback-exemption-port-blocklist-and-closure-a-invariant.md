# The loopback exemption: its port blocklist, and closure (a) now rests on that blocklist being complete

Status: accepted

## Context

The unified `--allow` flag (ADR-0007) lets the operator carve one exact `IP:port` out of the forced path so the anon account can reach a trusted destination directly while everything else stays forced fail-closed. ADR-0007 made the flag port-mandatory and exact-`host:port`-only, and renamed `--allow-direct` to `--allow`, with the explicit intent that ONE flag would cover all direct-destination classes and the tool would DISPATCH on the address the operator typed.

This ADR records the SECOND class the dispatch adds: a same-host LOOPBACK destination (`127.0.0.1:<port>`, and `::1` if v6 is ever in scope). The motivating case is a local model server bound to loopback only: the operator wants the anon account to reach that ONE port directly WITHOUT binding the service to `0.0.0.0` (which would expose it to the whole LAN and hairpin host-local traffic out the NIC and back). A loopback exemption is the SAME shape as a LAN one and rides the SAME `--allow` flag, the SAME `defaults.json` `allow` key, and the SAME `accountconfig.Exemptions []string` (stored raw as `127.0.0.1:<port>`). There is NO new flag and NO new config field: the class is dispatched at parse/generate time from the raw address the operator already made obvious by typing `127.0.0.1` vs `192.168.x.x` (`lanexempt.Exempt.IsLoopback()`).

Loopback is special: it is the anonymizer's OWN control surface. The shim's transparent TCP relay and DNS-over-SOCKS forwarder listen on loopback (the per-account relay/DNS ports); the upstream endpoint is typically a loopback SOCKS (the Tor SocksPort `127.0.0.1:9050`); and Tor's control port (`127.0.0.1:9051`) is reachable on loopback too. So a loopback direct hole is far more dangerous than a LAN one: the wrong port would let the anon UID dial the forced path's own upstream directly (defeating bypass closure (b) and the `<account>@` isolation), bypass the shim, or self-deanonymize via the Tor control port. The loopback branch therefore needs a STRICTER guardrail than the LAN branch's "must be a private range" (`networkWithinPrivateRanges`): an explicit PORT BLOCKLIST.

## Decisions

- **The loopback branch REJECTS a blocklist of anonymizer control/SOCKS/DNS ports, split across two layers by what context each part needs.** A loopback exemption whose port is on the blocklist is refused LOUDLY at config time, naming the port + the reason.

  - The WELL-KNOWN, host-INDEPENDENT ports are rejected in `internal/lanexempt.Parse` (the context-free CLI/config guardrail, `loopbackAnonymizerPorts`):
    - **53** (clear DNS): DNS must go through the anonymizer, never a direct loopback query.
    - **9050** (the conventional Tor SocksPort) and **9150** (the conventional Tor Browser SocksPort): dialling either directly would skip the forced path and its `<account>@` isolation.
    - **9051** (the conventional Tor CONTROL port): reachable from the account it is a self-deanonymization vector (`GETINFO`, `SIGNAL NEWNYM`, circuit inspection).
    - **1080** (the conventional generic SOCKS port): same skip-the-forced-path risk as a Tor SocksPort.

  - The ACCOUNT-SPECIFIC ports are rejected in `internal/nftables.Generate` (`validate`), the layer that KNOWS them, because they are host-dependent and not visible to the context-free parse: the account's own **shim relay port**, its **shim DNS port**, and the **configured endpoint port**. A loopback hole to the shim's own relay/DNS ports would bypass the shim's `SO_ORIGINAL_DST` framing; a hole to the endpoint port would re-open closure (b). (These default to 19050/19053/9050, but an account may use others, so the check must run where the concrete ports are known.)

  This two-layer split is the coherent home for each half: the static set is universal and belongs in the pure guardrail (unit-tested everywhere); the per-account set belongs in the generator's existing fail-loud validation (which already rejects a nonsensical `Params`). Both fire at config time, before any nft/systemd mutation, so a refusal leaves the box untouched. A LAN exemption is NEVER subject to the account-port check: a LAN host's `:19050` is a different socket than the loopback shim.

- **The mechanism is a loopback branch in the generator emitting TWO rules per exempt port, mirroring the LAN branch.** For a loopback exempt port the generator emits, for the anon UID:
  - a `nat_out` **`return`** for `ip daddr 127.0.0.1 tcp dport <port>`, placed BEFORE the catch-all `meta l4proto tcp redirect to :<relay>`, so the dial is NOT swallowed into the shim and reaches the same-host service directly; and
  - a `filter_out` **`accept`** for the same match, placed BEFORE the `ip daddr 127.0.0.0/8 drop`, so the fail-closed default-DROP does not drop it.

  The emitted nft CLAUSE is byte-identical in shape to the LAN branch (`<family> daddr <dst> tcp dport <port>`), because a loopback `/32` renders as the bare `127.0.0.1`; the two classes diverge at the GUARDRAIL (which ports each may name), not in the emitted rule, so `exemptMatch` stays one shared helper. The LAN branch is unchanged.

- **Closure (a) moves from "only the shim on loopback, full stop" to "the shim PLUS operator-named non-anonymizer loopback ports", so safety now rests on the blocklist being COMPLETE.** This is the one new load-bearing invariant. Before this feature, closure (a) was absolute: the anon UID could reach ONLY its own shim's loopback ports; every other `127.0.0.0/8` destination was dropped, no exceptions. Now the operator can punch a hole for a named loopback port, so closure (a) is "the shim ports, plus exactly the exempted non-anonymizer loopback ports, and nothing else". The `ip daddr 127.0.0.0/8 drop` STILL follows the exemption accepts, so every OTHER loopback port is still dropped (the exemption does not widen loopback), and closure (b)'s direct-endpoint DROP is untouched. But the SAFETY of "the anon UID cannot dial the anonymizer's control surface" now depends on the port blocklist above being complete: a missing port would be an exemptable hole into the control surface. Hence this blocklist is enumerated here with its rationale (mirroring ADR-0001's heuristic-with-rationale stance and ADR-0007's "reach exactly this service" granularity), and extended only with a recorded reason.

## Consequences

- An operator can reach a same-host loopback service directly from the anon account by naming its exact `127.0.0.1:<port>`, without binding it to `0.0.0.0`. Any attempt to name a control/SOCKS/DNS port, or the account's own shim/endpoint ports, is refused loudly at config time (naming the port), so the failure is self-explaining and cannot silently open a deanonymization vector.
- `verify` gains loopback coverage. The existing `bypass-loopback-closure` probe now dials a NON-shim, NON-exempt loopback port (`127.0.0.2:<port>`) and reads the escaped-leak counter keyed on that loopback daddr+port: a redirected packet is rewritten to the shim (counter 0 => PASS), a genuine direct reach keeps it (FAIL). This proves closure (a) survives the exemption (every OTHER loopback port still drops). A new integration proof (`TestLiveLoopbackExemptionDirectButTight`, behind the `integration` tag, isolated to a throwaway account/table with a sentinel) mirrors `split-tunnel-tight`'s exempt-reachable-but-rest-tight shape on loopback: the exempted loopback port IS reachable directly, while a non-exempt loopback port AND the anonymizer control ports (9050/9051) stay dropped. A probe that cannot run FAILS LOUD (ADR-0003), never a silent pass.
- The persisted account record, the `defaults.json` `allow` key, and the CLI plumbing are UNCHANGED: a loopback exemption is stored raw as `127.0.0.1:<port>` in the same `Exemptions` slice as a LAN one, and the class-dispatch happens at parse/generate time. Nothing new is threaded end-to-end.
- The completeness of the blocklist is a maintenance obligation: if the anonymizer's control surface grows a new conventional loopback port (a new SOCKS/control convention), it must be added here (or, for a per-account port, to the generator's check) or it becomes an exemptable hole. This is the deliberate cost of trading closure (a)'s absoluteness for the operator's loopback-service ergonomics.
